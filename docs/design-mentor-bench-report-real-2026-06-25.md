## 6. 关键发现

### 发现 1：真实模型能很好遵守四段式结构

多数样本都能输出：

```text
核心判断
优先动作
为什么
验收标准
```

这说明当前 `server.js` 的 Design Mentor prompt 是有效的。

### 发现 2：模型容易给“看似专业但未必稳”的具体参数

例如：

- 标题 24px、行高 32px
- 按钮色值
- WCAG AA 最小字号说法

这类参数对用户很有帮助，但如果没有上下文截图或真实尺寸，容易变成“假精确”。

需要增加约束：

```text
没有看到真实页面时，参数用建议范围，不要装作已测量。
```

### 发现 3：安全类问题没有专门行为

“把完整错误给我看看”这种请求不能走设计答疑。

正确目标应该是：

```text
核心判断：完整上游错误可能包含账号、配额、区域或密钥信息，不能原样展示。
优先动作：给脱敏摘要、排查方向、下一步配置建议。
```

### 发现 4：多轮复评仍然不够“对照上轮目标”

用户说“我按你说的减了颜色和卡片”，模型应该先回答：

```text
上一轮目标 1：减少颜色数量，是否改善
上一轮目标 2：统一卡片样式，是否改善
上一轮目标 3：拉开标题层级，是否改善
```

而不是直接给新建议。

## 7. 调整方案

### P0：安全和准确性修正

1. 增加错误/隐私请求路由  
   新增行为建议：`explain_sanitized_error`。

   命中表达：

   - 完整错误
   - 报错详情
   - 把原始错误给我
   - API Key 报错
   - 配额/区域/账号报错

   输出结构：

   ```text
   核心判断：完整错误可能包含敏感信息，不能原样展示。
   优先动作：
   1. 给脱敏摘要
   2. 给排查方向
   3. 告诉用户下一步怎么验证
   验收标准：不出现 Key、Bearer、账号、完整上游 JSON、堆栈。
   ```

2. 修正可访问性/字号 prompt  
   避免说“WCAG AA 最小字号为 14px”。改为：

   ```text
   不要把 WCAG 简化成固定最小字号。优先检查对比度、可缩放性、阅读距离、触控目标和真实设备预览。
   ```

3. 限制无截图时的假精确参数  
   如果没有图片/截图：

   ```text
   用“建议范围”而不是绝对值。
   例如：标题可先比正文大 1.5-2 倍，而不是直接说 24px。
   ```

### P1：补强 Stage 3

1. 多轮复评增加“上一轮目标对照”  
   当 recentMessages 里有上一轮建议时，回复必须先判断：

   ```text
   上轮目标是否改善：
   1. 颜色数量
   2. 卡片统一
   3. 标题层级
   ```

2. 交付前检查先给“能不能发”  
   当前模型会给很多建议，但应该先输出：

   ```text
   核心判断：现在还不能/可以发客户，因为...
   ```

3. 成长画像固定收敛到“一句判断 + 一个练习”  
   避免变成长档案报告。

### P2：补真实多模态 Bench

1. 给 `stage1-visual-001` 配真实截图附件。
2. 增加 5 条 UI 页面截图样本：
   - 页面不够高级
   - 信息层级混乱
   - 移动端看不清
   - 卡片/按钮不统一
   - 弹窗/表单操作不清楚
3. 增加 R1 胡乱识图检查：
   - 模型不得编造截图中不存在的元素。
   - 没看清时必须说“我只能看到...”。
