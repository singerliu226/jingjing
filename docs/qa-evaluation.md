- 项目详情：核心字段、自动回填入口、删除项目入口是否存在。
- 附件安全：图片 base64 只用于当次请求和预览，不长期写入项目状态。
- 服务端模型分流：普通文本走 `qwen-plus`，图片走 `qwen-vl-plus`。
- 核心记录：新项目、交付物、截止时间、反馈、风险、待办是否能整理。
- 设计咨询：字体/设计问题不应被误写成项目任务。
- 模型实体：`requirements`、`progressNote` 能进入项目上下文。
- 工作流：项目信息完整后能生成下一步任务。
- 真实接口：`/api/health`、`/api/intent`、文本 `/api/chat`、图片 `/api/chat`。

## 具体反馈

### 可用性

当前核心链路基本跑通：用户可以在中间输入需求/问题，系统能整理项目状态；右侧项目详情能作为上下文；图片上传已具备最小闭环。

但产品还不是“很顺手”的成熟工具。现在更像一个可工作的 MVP，离菁菁每天自然使用还有几处体验差距。

### 主要问题

1. 服务地址入口已强制  
   直接打开 `file://.../index.html` 时会显示服务入口面板，并阻断工作台交互。正式使用只引导访问 `http://localhost:4174/` 或部署后的线上地址。

2. 设计咨询分类仍偏粗  
   例如“标题字体怎么配比较好？”功能上不会误写项目，但可能被归为泛化的 `answer_design_question`，不是更细的 `recommend_typography_system`。这不影响当前体验，但会影响后续数据分析和能力标签沉淀。

3. 图片理解还只是“看图诊断”，不是完整审美 agent  
   现在能看图并指出问题，但还没有独立的视觉评分、修改优先级、对比版本、生成改稿 brief。建议下一阶段把图片能力拆成独立的“视觉诊断模块”，再考虑接即梦生成参考方向。

4. 文件能力还轻  
   目前文本文件只作为对话上下文，未做 PDF、PPT、PSD/AI 源文件解析。对平面设计师来说，后续更重要的是：设计稿截图、参考图、多版本对比，而不是泛文件管理。

5. 右侧任务编辑仍偏工程化  
   “状态、优先级、截止、细节”已经可用，但对小白设计师仍略像后台系统。更自然的形态应该是：小画桌给出“下一步建议”，用户只点“采纳/改一下/完成”。

## 下一阶段建议

优先级从高到低：

1. 继续打磨服务化使用说明  
   README 和部署文档需要明确：请打开 `http://localhost:4174/`；公网部署时配置允许的 Origin，必要时设置 `DESIGN_DESK_API_TOKEN`。

2. 增加“视觉诊断 agent”而不是马上接即梦  
   输入：图片 + 项目详情 + 用户问题。  
   输出：画面判断、三条最该改的问题、修改优先级、可执行改法、需要补充的信息。

3. 再接即梦作为“视觉方向生成 agent”  
   只在用户明确说“帮我生成参考方向 / 视觉小样 / 改稿方向图”时调用，不参与日常记录和诊断。

4. 把右侧任务改成建议卡  
   由 LLM 生成 1-3 条“今天先做”，用户点按钮采纳，而不是手动维护复杂任务表。

5. 增加版本对比  
   上传 V1/V2 两张图后，让小画桌说明“改了什么、哪里变好、哪里还弱”，这会非常贴合初级设计师复盘和作品集沉淀。
