# Design Growth Agent 改进方案与 Bench 标准设计

## 1. 改进目标

这个 agent 目前已经能帮助初级 UI 设计师整理反馈、拆行动项、降低动手阻力，但它还需要从“任务型助理”进一步变成“成长型设计 mentor”。

核心改进方向：

- 少给过程性信息，多给判断、重点和下一步。
- 少用固定模板，多根据真实语境生成设计反馈。
- 不只告诉用户“做什么”，还要让用户逐渐理解“为什么这样判断”。
- 把一次回复变成一个可跟随的设计判断过程，而不是一整块说明文。

目标用户是初级 UI/UX 设计师。理想体验不是“AI 帮我写了一段建议”，而是“它像一个资深同事一样，抓住重点、带我看问题、让我知道下一版怎么改”。

## 2. 产品改进方案

### 2.1 回复形态：从整块答案改为分步显影

现状问题：

- 大段卡片会让用户看不清重点。
- 过程性同步信息和真正建议混在一起。
- 用户难以判断第一步该做什么。

改进方案：

1. 将 assistant 回复拆成 3-5 个渐进块。
2. 第一块只放核心判断，例如“问题不是不够装饰，而是层级、留白和材质语言不统一”。
3. 第二块给 2-3 个优先动作。
4. 第三块解释判断依据，但控制长度。
5. 第四块给验收标准，例如“改完后截图缩到 25% 仍能看出主次”。
6. 过滤“我已整理、我同步了、我记录到”等过程句，只在需要时放入状态区。

建议回复结构：

| 层级 | 内容 | 用户感受 |
| --- | --- | --- |
| 结论 | 一句话判断设计问题 | 先知道重点 |
| 优先级 | 只给 2-3 个最该改的点 | 不被压垮 |
| 修改动作 | 明确到字体、间距、颜色、布局、组件状态 | 能马上动手 |
| 判断理由 | 用设计语言解释原因 | 增长审美判断 |
| 验收标准 | 如何判断改好了 | 能自查 |

### 2.2 回复策略：设计咨询优先走模型，确定性操作走本地

应继续缩短 `shouldKeepLocalReply/localOnlyBehaviors` 这类白名单。

适合本地模板的内容：

- 新建/完成/延期任务。
- 修改项目名称、截止时间、项目类型。
- 记录明确事实，例如版本号、客户确认结果。

应该走模型的内容：

- “高级感不够怎么改”
- “这页视觉有什么问题”
- “老板说太普通，我该怎么回应”
- “帮我复盘为什么这稿不成立”
- “我作为初级设计师哪里需要提升”

判断原则：

如果问题涉及审美判断、设计推理、表达策略、成长反馈，就不应只走本地模板。

### 2.3 Mentor 能力：建立设计诊断框架

agent 每次做 UI/UX 反馈时，应默认从以下维度看问题，但不要全部展示给用户：

| 维度 | 内部检查点 | 对用户展示方式 |
| --- | --- | --- |
| 信息层级 | 主标题、次要信息、行动入口是否清楚 | “主次不够拉开” |
| 布局节奏 | 留白、行高、模块间距是否稳定 | “页面呼吸感不足” |
| 字体系统 | 字号、字重、行距是否过多 | “字体层级需要收敛” |
| 色彩系统 | 主色、辅助色、灰阶是否过散 | “颜色数量过多，削弱高级感” |
| 组件一致性 | 按钮、卡片、标签是否统一 | “组件语言不一致” |
| 材质与质感 | 阴影、边框、背景、透明度是否克制 | “装饰比内容更抢” |
| 可用性 | 用户是否知道下一步做什么 | “视觉好看但操作入口弱” |
| 可访问性 | 对比度、可读性、触控目标是否足够 | “可读性风险” |

### 2.4 成长机制：从一次性建议变成持续成长档案

建议增加“设计成长画像”，记录用户反复出现的问题。

可记录字段：

- 常见弱项：层级、留白、配色、组件一致性、动效节奏、表达汇报。
- 当前阶段：入门、能独立改稿、能解释设计理由、能带方案。
- 最近 3 次反馈关键词。
- 下次优先练习项。
- 已改善项。

展示方式要轻，不要让用户觉得被考试。可以在周复盘或用户主动问时展示：

> 你最近三次问题都集中在“信息层级”和“留白节奏”。下一轮先别加装饰，先练习用字号、字重和间距把主次拉开。

### 2.5 交互工作流：形成“诊断-修改-验证”闭环

设计类任务建议固定成一个小闭环：

1. 诊断：指出最影响质量的 1-2 个问题。
2. 修改：给出具体修改动作。
3. 示例：必要时给一个 before/after 文案或布局示意。
4. 验证：告诉用户如何判断改好了。
5. 复盘：用户发回新版后，只比较本轮目标，不重新展开所有问题。

## 3. Bench 评测设计

### 3.1 评测对象

Bench 名称建议：

**DesignMentorBench：面向初级 UI/UX 设计师的成长型 Agent 评测集**

它评估的不是 agent 是否“说得多”，而是：

- 能否看懂设计问题。
- 能否抓住最关键的改动。
- 能否用初级设计师能理解的方式指导。
- 能否帮助用户形成设计判断力。
- 能否在多轮修改中持续推进，而不是每轮重新泛泛而谈。

### 3.2 借鉴的评测思想

参考文档里的方法，可以迁移三条核心原则：

1. **分阶段评测**  
   不把所有能力混成一个总分，而是按“理解诊断、指导共创、验证成长”三个阶段评。

2. **分类记分**  
   设计质量、行动可执行性、mentor 感、交互体验、可靠性分开打分。

3. **红线单列**  
   严重误导、胡乱识图、泄露敏感信息、未经确认大改用户文件等问题，不用平均分掩盖。

### 3.3 三阶段评测框架

| 阶段 | 名称 | 评测重点 | 激活指标 |
| --- | --- | --- | --- |
| Stage 1 | 理解与诊断 | 是否理解用户真实问题，能否抓住设计主要矛盾 | DQS、FSS、UCS、RGS |
| Stage 2 | 指导与共创 | 是否给出可执行、高质量、有教学感的设计建议 | AAS、VQS、MGS、IXS |
| Stage 3 | 验证与成长 | 是否能复盘改动效果，沉淀用户能力，持续推进 | VFS、LGS、MCS、DVS |

### 3.4 指标体系

#### Stage 1：理解与诊断

| 指标 | 含义 | 评分关注点 |
| --- | --- | --- |
| DQS 设计诊断质量 | 是否准确识别设计问题 | 能否指出层级、留白、字体、颜色、组件、可用性等真正问题 |
| FSS 重点筛选能力 | 是否过滤过程信息并突出重点 | 是否避免大段铺陈，只给当前最重要的 1-3 件事 |
| UCS 用户语境理解 | 是否理解用户身份和任务压力 | 是否按初级设计师的能力水平给建议 |
| RGS 可靠性与边界 | 是否避免臆测和误导 | 不虚构截图内容，不假装看过不存在的文件 |

#### Stage 2：指导与共创

| 指标 | 含义 | 评分关注点 |
| --- | --- | --- |
| AAS 行动可执行性 | 建议能否马上落地 | 是否具体到字号、间距、布局、颜色、组件状态或表达方式 |
| VQS 视觉/UX 专业质量 | 建议本身是否符合 UI/UX 常识 | 是否提升层级、可读性、一致性、交互效率 |
| MGS Mentor 成长感 | 是否帮助用户理解为什么 | 是否用简明设计理由提升判断力，而不是只给命令 |
| IXS 交互体验质量 | 回复是否像逐步引导 | 是否分条、渐进、重点清楚、没有压迫感 |

#### Stage 3：验证与成长

| 指标 | 含义 | 评分关注点 |
| --- | --- | --- |
| VFS 验证标准 | 是否给出改完怎么判断 | 是否有可检查的验收标准 |
| LGS 学习增益 | 用户是否更懂设计判断 | 是否沉淀一个可复用原则 |
| MCS 记忆与连续性 | 多轮中是否记得前文目标 | 是否围绕上轮目标评价，而不是重新泛泛批评 |
| DVS 交付价值 | 是否产出清晰结果 | 是否能形成清单、改稿方向、汇报话术或下一步计划 |

### 3.5 评分方式

每个子项采用 0-4 分：

| 分数 | 标准 |
| --- | --- |
| 0 | 缺失、错误或有明显误导 |
| 1 | 泛泛而谈，基本不可执行 |
| 2 | 部分相关，但重点不稳或行动不够具体 |
| 3 | 基本准确、清楚、可执行 |
| 4 | 准确、有洞察、能指导用户成长 |

统一映射为 0-10 分：

```text
Metric_Score = Mean(item_scores) / 4 * 10
```

阶段分：

```text
Stage_1 = Mean(DQS, FSS, UCS, RGS)
Stage_2 = Mean(AAS, VQS, MGS, IXS)
Stage_3 = Mean(VFS, LGS, MCS, DVS)
```

总质量分：

```text
Quality_Score = Mean(Stage_1, Stage_2, Stage_3)
```

成长效果单列：

```text
Growth_Efficacy = 用户后测理解分 - 用户前测理解分
```

说明：Growth_Efficacy 不直接混入 Quality_Score，避免把“用户本身进步快慢”和“agent 当轮表现”混在一起。

### 3.6 红线规则

以下问题单独记录，不被平均分掩盖：

| 红线 | 判定 |
| --- | --- |
| R1 胡乱识图 | 用户给了图，但 agent 编造图中不存在的元素 |
| R2 误导性专业建议 | 给出明显破坏可用性、可访问性或业务目标的建议 |
| R3 过程污染 | 把大量内部整理、同步、推理过程甩给用户 |
| R4 未经确认高风险改动 | 在用户未授权时大范围改文件、删内容或改核心逻辑 |
| R5 敏感信息泄露 | 暴露密钥、上游错误详情、用户隐私或内部配置 |
| R6 反成长反馈 | 用打击式、嘲讽式、居高临下方式评价初级设计师 |

红线处理：

- 出现 R1/R5：该样本直接失败。
- 出现 R2/R4：该样本最高 6 分。
- 出现 R3/R6：该样本最高 7 分。

### 3.7 样本集设计

建议第一版 Bench 做 80 条样本：

| 类型 | 占比 | 样例 |
| --- | --- | --- |
| 截图诊断 | 25% | “老板说不高级，这页怎么改？” |
| 模糊反馈转译 | 15% | “客户说有点乱，但没说哪里乱” |
| 视觉系统收敛 | 15% | 字体、颜色、卡片、按钮过多 |
| 交互/流程优化 | 10% | 表单、聊天、任务流、设置页不好用 |
| 多轮改稿 | 15% | 用户发初稿、agent 建议、用户发新版、agent 复评 |
| 汇报表达 | 10% | 帮初级设计师向老板解释改稿逻辑 |
| 成长复盘 | 10% | 分析用户最近几次设计短板 |

每条样本应包含：

- 用户输入。
- 可选截图/Figma/HTML 上下文。
- 用户身份设定。
- 任务阶段。
- 期望输出类型。
- 评分项。
- 红线检查项。
- 参考答案要点，而不是唯一标准答案。

### 3.8 示例样本

#### 样本 A：不够高级

用户输入：

> 老板说这个页面不够高级，我应该怎么改？

期望表现：

- 先判断“不高级”可能来自层级、留白、字体、色彩、装饰失控，而不是直接加阴影和渐变。
- 给 3 个优先动作。
- 提醒用户先收敛变量，再加质感。
- 给可验证标准。

低分表现：

- 只说“加高级配色、加动画、加玻璃拟态”。
- 一次给十几条建议。
- 没有说明为什么。

#### 样本 B：用户要求“多一点酷炫”

用户输入：

> 能不能加点动效，让页面看起来更酷？

期望表现：

- 先判断动效是否服务任务。
- 建议只在状态变化、重点进入、操作反馈处加动效。
- 给出时长、缓动、触发位置。
- 不建议满屏装饰动效。

#### 样本 C：多轮复评

第一轮用户：

> 这是我的任务管理页，帮我看看哪里乱。

第二轮用户：

> 我按你说的减了颜色和卡片，你再看看。

期望表现：

- 第二轮先对照上一轮目标复评。
- 只评价本轮是否改善。
- 不重新展开一套全新建议。
- 给下一步更细的打磨方向。

### 3.9 评测报告输出

每次 Bench 输出应包含：

- 总质量分 Quality_Score。
- 三阶段分：理解诊断、指导共创、验证成长。
- 红线结果。
- Top 3 强项。
- Top 3 短板。
- 最差样本回放。
- 对产品的改进建议。

推荐报告格式：

| 模块 | 输出 |
| --- | --- |
| 总体结论 | 当前更像任务助理 / 合格设计助理 / 成长型 mentor |
| 阶段得分 | Stage 1 / Stage 2 / Stage 3 |
| 红线 | 是否出现胡乱识图、泄露、过程污染等 |
| 体验问题 | 是否大段、是否模板化、是否缺少下一步 |
| 能力问题 | 诊断不准、建议太泛、缺少教学解释等 |
| 改进优先级 | P0/P1/P2 |

### 3.10 达标线

| 等级 | 分数 | 定义 |
| --- | --- | --- |
| 不可用 | < 6.0 | 建议泛、误判多，用户难以行动 |
| 可用助理 | 6.0-7.4 | 能整理问题，但 mentor 感弱 |
| 好用设计助理 | 7.5-8.4 | 能抓重点、给动作，适合日常改稿 |
| 成长型 mentor | >= 8.5 | 能连续指导、解释判断、帮助用户形成方法 |

上线建议：

- Quality_Score >= 7.5。
- Stage 1、Stage 2 不低于 7.0。
- R1/R5 红线为 0。
- “过程污染”样本失败率低于 5%。
- 多轮复评样本得分不低于 7.5。

## 4. 迭代路线图

### P0：先让它变得更好读

- 回复分块渐进展示。
- 过滤过程性信息。
- 设计咨询走模型，本地模板只处理确定性操作。
- 每次最多给 3 个优先建议。

### P1：让它更像 mentor

- 引入设计诊断框架。
- 增加“为什么这样改”的轻量解释。
- 增加验收标准。
- 支持用户发新版后做对照复评。

### P2：让它能陪伴成长

- 建立用户设计成长画像。
- 做周复盘和能力雷达。
- 记录反复出现的设计短板。
- 根据用户阶段调整建议粒度。

### P3：Bench 自动化

- 固化 80 条样本。
- 接入自动评分和人工复核。
- 每次改回复策略后跑回归。
- 把红线样本作为上线门禁。

## 5. 一句话产品原则

这个 agent 不应该追求“回答得完整”，而应该追求：

> 每次只抓住最影响设计质量的那个问题，用初级设计师听得懂的方式，带他改一小步，并让他知道为什么这一小步是对的。
