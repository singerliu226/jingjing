# 菁菁小画桌 LLM First 新版本计划

## 目标

把菁菁小画桌从“规则驱动的工作记录工具”调整为“简洁、好用、由大模型理解用户意图的设计工作助手”。

新版不再靠不断增加正则和行为分支来覆盖问题，而是让大模型承担理解、分类、追问和表达，本地代码负责可靠存储、任务更新、风险兜底和界面呈现。

## 当前问题

1. 行为识别仍然太像规则系统
   - 已经有 `/api/intent`，但 Core 里仍有大量规则判断。
   - 规则越多，新增场景越容易互相误伤。
   - 用户说法稍微变化，就需要继续补规则。

2. 回复链路不够统一
   - 前端先本地生成回复，再决定是否调用千问。
   - `shouldKeepLocalReply` 里保留了大量“本地回复优先”的行为。
   - 结果是用户以为在和 AI 对话，但很多时候看到的是固定模板。

3. 产品界面仍然有工具感
   - 工作台、项目小纸条、最近记录、任务清单都存在，但主路径还不够简单。
   - 用户进来应该只知道一件事：把今天的需求、反馈、进度丢给小画桌。
   - 其他能力应该作为结果自然出现，而不是入口堆在眼前。

4. 测试主要验证规则命中
   - 当前测试大量断言某句话命中某个固定 behavior。
   - 新版需要更多验证“模型结构化结果是否能安全驱动状态更新”。

## 产品原则

1. 一个主入口
   - 用户只需要输入自然语言。
   - 不要求她理解项目管理、prompt、作品集、知识库这些概念。

2. 模型先理解，本地再执行
   - 模型判断用户意图、抽取结构化信息、决定是否追问。
   - 本地只执行经过校验的结构化动作。

3. 规则只做护栏
   - 保留日期解析、文件状态、任务状态、数据校验、模型失败兜底。
   - 不再用规则扩展新的设计问题类型。

4. 回复必须可执行
   - 不说空泛审美词。
   - 每次回复尽量包含：已理解、下一步、需要确认、风险。

5. 界面默认轻
   - 首页只突出今日要做、对话输入、当前项目小纸条。
   - 作品集、知识库、复杂检查清单作为结果沉淀，不抢主流程。

## 目标架构

### 1. LLM 意图层

接口：`POST /api/intent`

模型输出不只返回 behavior，而是返回更完整的结构化意图：

```json
{
  "intent": "record_feedback",
  "confidence": 0.86,
  "summary": "主管反馈海报颜色太暗，需要更年轻",
  "entities": {
    "projectName": "",
    "source": "主管",
    "dueDate": "2026-06-13",
    "deliverables": [],
    "feedback": {
      "raw": "颜色太暗，要更年轻一点",
      "action": "提高明度和活力感",
      "reason": "当前视觉情绪偏沉"
    }
  },
  "missing": ["尺寸", "交付格式"],
  "nextAction": "先确认尺寸和投放平台，再调整配色明度"
}
```

### 2. 本地执行层

Core 只负责：

- 校验模型返回的 intent 是否在允许列表内。
- 校验 entities 是否合理。
- 更新项目、反馈、任务、风险、消息。
- 在模型不可用或低置信时调用本地兜底。

### 3. 回复生成层

接口：`POST /api/chat`

回复不再被大量本地模板截断。原则：

- 工作记录类：本地先更新状态，再让模型基于更新后的状态生成自然回复。
- 高风险动作：本地必须展示可靠兜底，例如任务已保存、反馈已记录。
- 模型失败：显示简洁失败提示和本地整理结果。

### 4. UI 呈现层

界面只展示三件事：

- 今天要做：由结构化任务生成。
- 最近对话：用于输入和查看 AI 整理结果。
- 项目小纸条：作为当前项目上下文，不做复杂表单系统。

其他能力收敛为：

- 交付检查：在交付风险出现时展示。
- 作品集沉淀：项目完成后自动提示。
- 知识库：不作为主导航入口，先作为回答中的引用或折叠内容。

## 版本路线

### Phase 1：统一意图结构

目标：让 `/api/intent` 从“只返回 behavior”升级为“返回结构化意图包”。

工作：

- 定义 `LLMIntent` 数据结构。
- 让服务端 prompt 输出 intent、entities、missing、nextAction。
- 客户端调用 `/api/intent` 时必须传入当前日期，模型解析“今天、明天、下周一”等相对时间时以该日期为准。
- Core 新增 `applyModelIntent`，优先使用模型实体更新状态。
- 保留 `analyzeInput` 作为兜底，不再作为主路径。

验收：

- 同一句话如果模型返回结构化反馈，本地能直接记录反馈和任务。
- 模型低置信时自动回退本地规则。
- 未知 intent 不会破坏状态。

### Phase 2：减少本地回复模板

目标：让用户明显感觉是在和 AI 助手对话，而不是看固定模板。

工作：

- 缩短 `shouldKeepLocalReply` 列表，只保留强状态类动作。
- 设计 `localUpdate + modelReply` 链路。
- 对话体验采用“本地已整理 + 模型建议”的组合，不让模型回复覆盖可靠的本地状态更新。
- 非状态类设计咨询直接模型优先展示，避免先暴露本地长模板再追加模型回复。
- 普通对话不使用固定五段式；只有项目小纸条工作流继续使用固定结构。
- 模型回复必须基于最新 state，不允许编造已完成结果。
- 删除或降级部分过细的本地设计建议模板。

验收：

- 设计咨询类问题默认进入模型回复。
- 工作记录类问题仍可靠更新本地任务和反馈。
- 模型失败时有简洁兜底，不阻塞用户。

### Phase 3：简化首页主路径

目标：用户一进来只需要知道“把今天的事告诉小画桌”。

工作：

- 首页主视觉收敛成：今日要做、输入框、当前项目。
- 最近记录改为真正的对话区，扩大阅读空间。
- 项目小纸条只保留关键字段：项目名、类型、截止、目标、交付物。
- 作品集和知识库从主导航弱化为自动沉淀入口。

验收：

- 首屏不出现多个同等权重的功能入口。
- 输入和今日任务是最明显的两个区域。
- 右侧/下方信息不遮挡、不拥挤。

### Phase 4：把规则库瘦身成护栏

目标：停止继续堆规则，让规则只服务可靠性。

工作：

- 梳理 `detectBehavior` 中的设计咨询类规则。
- 能交给模型判断的行为从本地优先改为模型优先。
- 保留稳定规则：日期、状态、交付物、项目字段、风险校验。
- 给每个保留规则标注原因。

当前进展：

- App 主输入已切换到 `localMode: "guardrail"`。
- 护栏模式只在本地识别稳定工作动作，例如今日计划、截止变更、项目字段、反馈记录、完成/等待状态、交付检查。
- 字体、配色、品牌一致性、版式诊断等细分设计咨询不再由 App 主路径的本地规则优先判断；模型返回结构化 intent 时，再由 Core 执行对应行为。
- 旧的细分规则暂时保留在 Core 中作为兼容兜底和迁移期测试对象，后续再按评估集逐步删除或下沉。

验收：

- 新增设计问题不需要新增正则即可被模型处理。
- 本地规则数量下降，职责更清楚。
- 测试从“规则命中”转向“结构化意图驱动状态正确”。

### Phase 5：建立真实使用评估集

目标：用真实初级设计师表达来验收，而不是只靠理想句子。

工作：

- 建立 `intent-evals` 样例集。
- 覆盖需求记录、反馈记录、设计求助、截止变更、交付检查、作品集沉淀。
- 每条样例记录期望 intent、关键实体、应该追问什么。
- 后续每次改 prompt 或规则都跑一遍。

验收：

- 先建立不少于 30 条种子样例，再扩展到至少 50 条真实表达样例。
- 关键行为识别准确率达到可接受水平。
- 错误样例能反向推动 prompt 或 UI 调整。

## 接下来执行顺序

1. 先做 Phase 1
   - 不改 UI。
   - 不新增设计问题规则。
   - 只把模型意图结构补完整，并让 Core 能吃结构化结果。

2. 再做 Phase 2
   - 调整回复链路。
   - 降低本地模板存在感。

3. 然后做 Phase 3
   - 基于新的交互逻辑重新简化界面。

4. 最后做 Phase 4 和 Phase 5
   - 清理规则。
   - 建立评估集，防止之后再散。

## 暂停新增的事项

在 Phase 1 和 Phase 2 完成前，暂时不做：

- 新增某一个具体设计问题的正则识别。
- 新增知识库入口。
- 新增作品集页面功能。
- 新增复杂项目管理视图。
- 继续扩大 `shouldKeepLocalReply` 白名单。

## 判断新版本是否更好的标准

1. 用户输入更自由
   - 不需要按固定句式说话。

2. 回复更像助手
   - 有上下文、有判断、有下一步，而不是模板堆砌。

3. 界面更轻
   - 主路径清楚，入口减少。

4. 状态仍可靠
   - 任务、反馈、项目、风险不会因为模型输出不稳而乱掉。

5. 迭代更稳
   - 后续新增能力先进入模型 prompt 和评估集，而不是先写新规则。
